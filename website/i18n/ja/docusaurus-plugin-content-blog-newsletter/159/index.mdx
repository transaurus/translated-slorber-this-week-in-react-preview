## ⚛️ React {#react}

[![Bun 1.0](/emails/issues/159/bun.jpg)](https://bun.sh/blog/bun-v1.0)

[**Bun 1.0**](https://bun.sh/blog/bun-v1.0)

今週は見逃せないニュースです：Bunが安定版v1.0としてリリースされました。このニュースは直接Reactに関連するものではありませんが、いずれすべてのReact開発者に影響を与えるでしょう。

Bunは、WebkitをベースにZigで書かれたJavaScriptとTypeScriptのための完全なツールチェーンです。現在使用している多くのフロントエンドツール（Node.js、npx、Babel、esbuild、swc、webpack、Jest、Vitest、npm、Yarn、pnpmなど）に対する超高速な代替手段です。ベンチマークは非常に印象的で、間違いなく本物です。

既存のツールとの互換性とESM/CJSの混合サポートにより、Bunは簡単にドロップイン代替として導入できます。Bunは一般的にコードを変更せずにそのまま実行可能です（そうでない場合は[🐦バグと見なされます](https://twitter.com/jarredsumner/status/1700497869160268155)）。Next.js、Remix、AstroなどのReactフレームワークは既にサポートされています。npm/Yarn/pnpmやJest/Vitestの代替として使用するのは良い導入点であり、リスクも少ないでしょう。

## 📱 React-Native {#react-native}

[![Static Hermes](/emails/issues/159/hermes.jpg)](https://speakerdeck.com/tmikov2023/static-hermes-react-native-eu-2023-announcement)

[**Static Hermes**](https://speakerdeck.com/tmikov2023/static-hermes-react-native-eu-2023-announcement)

Static Hermesは先週React-Native EUで発表されました。私は現地にいなかったため、動画がYouTubeに公開される前にスライドを読んで理解した内容を説明します。

Static Hermesは実験的なプロジェクトで、TypeScript（またはFlow）コードの一部をネイティブコードにコンパイルするオプションを提供します。これにより、Hermesの通常のバイトコードインタプリタモードと比べて大幅なパフォーマンス向上（10～20倍）が得られます。また、TypeScriptコードから直接ネイティブAPIと統合できるようになります（ゼロコストのFFI、JSIラッパー比15～80倍高速）。

これを有効にするには、まずTypeScriptを「健全な」言語にする必要があります。Static HermesはJavaScriptのセマンティクスを変更し、宣言された型が実行時の型と一致することを保証します。このスライドが概念をよく説明しています：

[![Static Hermes makes TypeScript sound](/emails/issues/159/hermes2.jpg)](https://speakerdeck.com/tmikov2023/static-hermes-react-native-eu-2023-announcement?slide=14)

非常に興味深いアイデアで、将来的にはReact-Nativeエコシステム以外にも影響を与える可能性があるかもしれません。
