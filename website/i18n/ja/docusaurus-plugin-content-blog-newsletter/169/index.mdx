## 💸 Sponsor {#sponsor1}

[![Nlux ― 会話型AIウェブアプリを作成する新しいReact / JSライブラリ](/emails/issues/169/nlux.png)](https://nlux.ai/?c-twr1-h)

**[Nlux ― 会話型AIウェブアプリを作成する新しいReact / JSライブラリ](https://nlux.ai/?c-twr1-h)**

[NLUX](https://nlux.ai?c-twr1-p1) *(Natural Language User Experienceの略)* は、ChatGPTのような強力な大規模言語モデル（LLM）をウェブアプリやウェブサイトに簡単に統合できる新しい[オープンソース](https://github.com/nluxai/nlux)のReact / Javascriptライブラリです。わずか数行のコードで、会話型AIの機能を追加し、お気に入りのLLMと対話できます。

主な特徴 🌟：

* **数分でAIチャットインターフェースを構築** - わずか数行のコードで高品質な会話型AIインターフェースを実現。
* **Reactコンポーネント＆フック** - UI用の`<AiChat />`と簡単な統合用の`useAdapter`フック。
* **柔軟なLLMアダプター** - `ChatGPT`と`HuggingFace`、そして任意のLLM用に**独自のアダプターを作成**するためのAPI。
* **カスタマイズ可能なチャットペルソナ** - ボットとユーザーの画像、名前、説明を設定してパーソナライズされた対話を実現。
* **ゼロ依存関係** - LLMフロントエンドライブラリを除き、軽量なコードベースで依存関係なし。

詳細、例、ドキュメントは[Nlux.ai](https://nlux.ai?c-twr1-p2)でご覧ください。

## ⚛️ React {#react}

[![React.useとReact.useMemoの未来 - コンテキストセレクターの強力な代替手段](/emails/issues/169/useMemo.jpg)](https://interbolt.org/blog/react-use-selector-optimization/)

**[React.useとReact.useMemoの未来 - コンテキストセレクターの強力な代替手段](https://interbolt.org/blog/react-use-selector-optimization/)**

Colinは、Reactコンテキストがパフォーマンス問題を引き起こす可能性があること、コンテキストセレクターAPIの不足、そしてDaishi Katoによるユーザーランド実装[use-context-selector](https://github.com/dai-shi/use-context-selector)を活用してアプリのパフォーマンスを最適化した方法について説明しています。

最も興味深い部分は、[最近のTwitterディスカッション](https://twitter.com/sebastienlorber/status/1741218919732429108)の優れたまとめです。Dan Abramovは、コンテキストセレクターがうまく構成できないことを説明しました。Andrew Clarkは、`useMemo()`を拡張し、その内部で他のフックを使用できるようにする計画を共有しました。これは、[任意のUsableタイプ](https://github.com/acdlite/rfcs/blob/first-class-promises/text/0000-first-class-support-for-promises.md#other-usable-types)（現在はコンテキストとプロミスですが、将来的にはObservablesやストアも含まれるかもしれません）とuseフックで動作しますが、[サードパーティのフック](https://twitter.com/BenJ_Ayc0/status/1742499230525067408)とも互換性があります！これにより、細かい選択問題がパフォーマンス良く構成可能な方法で解決され、今後のReact Forgetコンパイラとも互換性があります。

[![2つのReact](/emails/issues/169/2reacts.jpg)](https://overreacted.io/the-two-reacts/)

**[2つのReact](https://overreacted.io/the-two-reacts/)**

Dan Abramovによる素晴らしい記事で、コードをローカルで実行する（即時フィードバック）理由と、リモートで実行する（データソースに近い）理由の両方を示しています。彼は新しいReactの公式を提案しています：`UI = f(data, state)`。ここで`f`は、2つの異なる環境で実行されるあなたのReactアプリ全体を表します。

これは、この問題を解決するためにReact Server Componentsを発明するかどうかを暗に問う素晴らしい質問で締めくくられています。

> Reactの優れた点を保ちつつ、コンポーネントをあなたのコンピュータと私のコンピュータの間で分割する方法はあるでしょうか？2つの異なる環境からのコンポーネントを組み合わせてネストできるでしょうか？それはどのように機能するでしょうか？
