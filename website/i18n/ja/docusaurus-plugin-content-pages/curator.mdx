# This Week In React - キュレーターの仕事

私は当初、React開発者がエコシステムの最新情報をキャッチアップできるようにするキュレーションニュースレターとして[This Week In React](https://thisweekinreact.com/)を作成しました。

長期的な目標は、ReactとReact Native開発者のための真のコミュニティ主導のメディア（CSS-Tricksのような）になることです。単なる週刊メールのリンク集以上のものにしたいと考えています。

そのためには委任が必要で、週刊号の制作を手伝ってくれるニュースレター協力者を募集したいと考えています。

これは**本物の仕事**です：報酬が支払われ、**信頼性**が求められ、**1通のメールに14-18時間の作業時間**を割く意思が必要です。

## This Week In Reactにとってのメリット

この募集における私の主な目的は：

* バーンアウトを避けるため、私が休暇を取る際の信頼できるバックアップを持つこと
* 年間50号以上でメディアをプロフェッショナルに見せること
* メディアをよりコミュニティ主導にすること
* 著者をローテーションさせることでメディアをより多様にすること
* メディアをさらに発展させるための時間を確保すること
* ニュースレターの品質を高いまま維持すること

私はこれをほぼ独力で5年間続けてきましたが、最初は楽しかったことも長期的には負担になる可能性があります。強制されるのではなく、キュレーションニュースレターを発行したいときに選択できるようにしたいです。プロジェクトが最終的に私を超えて存続できるように、自分自身を方程式から外したいと考えています。

## ゲストキュレーターにとってのメリット

得られる主な利点：

* 各号ごとに**適切な報酬**が支払われる
* **個人ブランディング**と会社の**認知度向上**に貢献できる（[ゲストキュレーターが執筆した号の例](https://thisweekinreact.com/newsletter/206)）
* コミュニティに還元し、有用なプロジェクトに貢献できる

斬新なアイデアがあれば、得られるメリットについて話し合うことにオープンです。ウィンウィンの関係であれば。

## 理想的な候補者

私が探しているのは：

* **真剣に取り組み**、**非常に信頼でき**、**長期的**にこの仕事をする意思がある人
* 単一のメールに**完全な責任**を持てる人。覚えておいてください：私は休暇中で、送信ボタンを押すだけの存在です！あなたは独りで、私の立場に立つことになります。
* **ReactとReact Nativeの両方のニュース**についてコメントでき、興味深い**上級者レベルの洞察**を提供できる人
* 月に1回、**1通のメールに約16時間**を費やせる人。本当に**冗談ではありません**、これは副業としてできるものではなく、スケジュールに**その時間を確保**する必要があります。
* 様々なソース（RSS、X、有機的...）から興味深いリンクを見つけるための私のキュレーションプロセスに従える人
* ニュースレターを定期的に読み、対象読者を理解している人
* 私の編集方針とフォーマットガイドに従える人
* 英語で適切かつ簡潔にコミュニケーションを取れる人（完璧である必要はなく、必要に応じてLLMを使用しても構いません）。私はネイティブスピーカーではないので問題ありません。

はい、私は5本足の羊を探しているようなものです😅

理想的な候補者は単一の人物ではなく、貢献者を2人提供できるReactサービス会社かもしれません。私にとっての利点：

* 彼らは通常、長期的なマーケティングビジョンを持って真剣に取り組みます。
* 開発者を管理し、スケジュールに適切な時間を確保する方法を知っています。
* 号の責任を持ち、自社の評判を気にします。
* React WebとReact Nativeの両方をカバーする異なる専門知識を持つ安定したデュオを提供できます。
* 休暇を取ったり、貢献をやめたりした場合のバックアップを見つけることができます。

個人の貢献者にもオープンですが、企業の方が適していると考えています。私はすでに[Theodo Apps](https://apps.theodo.com/)と成功裏に協力しています（[例](https://thisweekinreact.com/newsletter/206)）。

## オンボーディング

これまでのところ、オンボーディングは以下の通りです：

* 自己紹介（または所属企業の従業員紹介）を行い、なぜ適任だと思うかを説明する。
* 有償のテストを実施し、事前に[ニュースレターカレンダー](https://thisweekinreact.com/calendar)から2回分のテスト発行を割り当てる。
* オンboarding通話を行い、プロセスを説明する。
* リンク収集用のキュレーションツール（RSSリーダー）へのアクセス権を付与する。
* 1回目のテスト発行では共同作業を行い、プロセスと細かなニュアンスを理解してもらう。
* 2回目のテスト発行では完全なオーナーシップを委譲。質問への回答とフィードバックは行うが、大部分を自律的に作業する。
* 作業内容をレビュー。完璧ではないが、送信するメールに恥ずかしさを感じなければ成功で、今後改善が見込める。
* 他の担当日を定義可能。

## プロセス

**要約**: ニュースレターのキュレーションプロセスは通常月曜日に開始し、水曜日夕方（~17時UTC/~9時PST）に送信。水曜日までにメールが100%完成している責任を負う。総作業時間は約16時間で、2名の開発者間で並行化可能。

**主要ステップ:**

* 数週間前に担当号を計画
* X、RSS、ウェブ、有機的ソースからおおよその興味深いリンクを収集
* これらのリンクを共同編集可能なGoogleドキュメントに追加
* リンクをレビューし、フォーマット調整と不要なものの選別
* 特に興味深いリンクへのコメント付与
* 見出し記事の選定
* ニュースに関連したミームの選定（可能であれば）
* React/RNキーワードを含むタイトル作成
* イントロテキストの作成
* MDX形式への変換
* Gitへのプッシュとオンライン版の公開
* SNSでのニュースレター宣伝協力

**私が責任を保持する部分:**

* ニュースレタースポンサーと広告配置の管理
* ウェブ版からメール版への変換と送信実行

**要調整事項:**

* X（旧Twitter）スレッドまたは記事の公開
* Bluesky、Reddit、LinkedIn等への投稿

数ヶ月前に[公開カレンダー](https://thisweekinreact.com/calendar)を確認し、担当日を決定。日付が決まると、あなたがその号の予定著者となり、**約2日分の作業時間**をカレンダーに確保する必要がある。

## キュレーションプロセス

情報収集プロセスは比較的予測可能:

* この[X時系列タイムライン](https://slo.im/x/feed)で過去7日間をスクロール（リダイレクト`https://slo.im/x/feed`をブックマーク推奨）
* 500以上のRSSフィードへのアクセス権取得

独自の方法（有機的または体系的）で興味深い情報を発見することも可能。未追跡の有益な情報源を見つけた場合、追跡を開始する必要がある。

## 編集方針

**基本理念:**

* **中級/上級React・React Native開発者**向けだが、React関連以外のリンクも含む。
* **新鮮なコンテンツ**。1ヶ月前の情報は既に古く、既にカバー済みの可能性。
* 未来の動向を理解するための**弱いシグナル**。
* 新リリース、新プロジェクト、新ドキュメントページなどの**イベント駆動型コンテンツ**。
* React npmパッケージの探索ツールではなく、**エコシステムの脈動**を伝える。

**優先すべき内容:**

* **単一のアイデア**に焦点を当て、概念を**深く**解説するコンテンツ。
* **Reactコア技術**に焦点を当て、広範な開発者層に適用可能な内容。
* **斬新的**、**革新的**、または**過小評価されている**技術を紹介するコンテンツ。
* 適切に提示され影響力のある**ニッチトピック**や**弱いシグナル**を扱うコンテンツ。
* インタラクティブ記事など、**卓越したプレゼンテーション**と優れた**読了体験**を提供するコンテンツ。

**避けるべき内容:**

* AI生成コンテンツ、明らかなSEO目的の記事、リスト形式の低品質記事は除外する。
* ペイウォールの裏にあるコンテンツ、認証が必要なもの、読みにくいユーザー体験を提供するコンテンツは除外する。
* 初心者向けの基本的な内容、ステップバイステップのチュートリアル、公式ドキュメントの言い換えにすぎない付加価値のないコンテンツは除外する。
* 有料製品の使い方を教えるコンテンツ（そのためにはスポンサーシップがある）は除外する。ただし、Reactのコアについて価値ある内容を教えつつ、過度に宣伝的でない場合は例外とする。
* ほとんどのUIライブラリやボイラープレートは除外する。数が多すぎるため。例外は、高い注目を集めている、長期的なメンテナンスが約束されている、または権威あるソースから提供されている場合。これらの「ローンチ」はカバーしないが、v10など主流になりコミュニティに価値を証明した場合は取り上げる。
* 弱すぎる弱シグナル: 理解できない、簡単な言葉で影響を説明できない、またはごく少数の人しか恩恵を受けられないと思われるリンクは時期尚早か、含める価値がない可能性がある。
* X/Twitterのスレッドや会話のリンクは可能な限り避ける: サインアップが必要。外部リンクを優先する。ただし、すべての情報やリンク、動画デモを含む個別のツイートは問題ない。

例外的に、優れたプレゼンテーション努力があるコンテンツは常に考慮される。例えば、基本的なReactの概念を説明するインタラクティブ記事は問題ない: これは上級開発者（ターゲットオーディエンス）がジュニアの同僚と共有するのに適した優れたリソースである。

## コンテンツ

ニュースレターには多くのリンクが含まれる。Reactエコシステムが活発なためである。
リンクをさらに厳選する代わりに、情報密度が高くても読みやすい体験を提供するよう最適化する。
コンテンツは一貫性があり、スキャンしやすく、可能な限り情報を「インライン化」する必要がある。
リンクをクリックさせてがっかりさせるようなことは避けたい。
ニュースレターを読むだけで、リンクをクリックしなくても価値を提供できるようにする。

### セクション

ニュースレターには5つの主要セクションがある:

* イントロ: ニュースの簡単なまとめ
* React: すべてのReact開発者、特にReact Web開発者に関連するコンテンツ
* React Native: React Native開発者に関連するコンテンツ
* その他: TypeScript、CSS、TC39、HTML、ツーリングなど、より広範なJSオーディエンスに関連するコンテンツ
* 楽しみ: 1-3つのミームまたは面白いスクリーンショット。Reactに関連するもの（内部ジョーク）や今週のニュースが好ましい。

### 見出し

通常、以下のように選ぶ:

* Reactの見出し1つ
* React Nativeの見出し1つ

各見出しには通常、トップバナー画像がある。
視覚的に意味のあるもの（コードのスクリーンショットなど）を選ぶのが好ましいが、美しいソーシャルカードでも構わない。

セクションに明確な見出しがない場合や2つある場合もある。
「その他」セクションの見出しを選ぶこともある。

### フォーマット

ほとんどのリンクは以下の規則に従う:

* \<絵文字> [リンクタイトル](https://example.com): 任意の主観的なコメント。

タイトルが長すぎる、クリックベイト的、または大文字小文字が不自然な場合は、すべてのタイトルが一貫性を持つようにリファクタリングしても構わない。

リリースの場合、「v」とパッチ番号を省略する（3.1、v3.1.9ではない）。主要リリースのハイライトキーワードを直接リンクに含め、ダッシュ「-」で区切る。ハイライトする内容がないパッチリリースは含めないことが多い。リンクの「中身」は「客観的」な部分（ハイライト）、「外側」は主観的なコメント（ある場合）。例:

* 📦 [Astro 5.2 - Tailwind 4サポート、末尾スラッシュリダイレクト、Reactストリーミング無効化オプション](https://astro.build/blog/astro-520/): ストリーミングはランタイムCSS-in-JSライブラリと互換性がない可能性があります。
* 📦 [Nextra 4.0 - App Routerを採用](https://the-guild.dev/blog/nextra-4)
* 📦 [React Scan 0.1.2 - 再レンダー履歴、コンポーネントツリー](https://github.com/aidenybai/react-scan/releases/tag/v0.1.2)

初めて紹介する比較的新しいプロジェクトの場合、バージョン番号を省略し、キーワードをGitHubリポジトリの説明やランディングページ、SEOタグからインスパイアされた客観的なプロジェクト説明に置き換えることができます：

* 📦 [Polygen - WebAssemblyモジュールを静的に、解釈せずにアプリ内で実行](https://github.com/callstackincubator/polygen): Callstackのプレビュー版パッケージで、WasmをCに変換しJSI経由で呼び出すことで優れたパフォーマンスを実現。
* 📦 [React Three Handles](https://pmndrs.github.io/xr/docs/handles/introduction): React XRインタラクションを構築するための便利な新パッケージ。
* 📦 [mdx-prompt - JSXで構成可能なLLMプロンプト](https://edspencer.net/2025/2/3/mdx-prompt-composable-prompts-with-jsx)

以下は一般的に使用する絵文字パレットです：

* 💸: スポンサー
* 👀: プレビュー、PR、弱いシグナル
* 💬: RFC、コミュニティディスカッション
* 📊: 調査、ベンチマーク
* 💡: 興味深いアイデア
* 📖: ドキュメント
* 🦋: Blueskyリンク
* 🐦: Twitter/Xリンク（可能なら避ける）
* 🎨: ビジュアルデモ（X以外で投稿）
* 📅: イベント、カンファレンスパートナーシップ
* 📜: 記事
* 📦: リリース、npmパッケージ、プラグイン、リポジトリ
* 🔗: リソース
* 🎥: 動画
* 🎙️: ポッドキャスト

通常、この「絵文字の順序」をほぼ尊重し、一貫した読み順を実現します：

* ヘッドライン
* 弱いシグナル

* メディアリソース

### コメント

一般的に従うべきルール:

* すべてにコメントする必要はありません。
* 特定の絵文字に対して、コメントする内容は通常最初に表示されます。
* 誤ったコメントをするよりは、コメントしない方が良いです。
* コメントは簡潔で要点を押さえたものにします。
* シンプルな言語を使用します。
* 不要な飾り言葉は追加しないでください。
